## 🎯 今日核心要点

### 为什么 PEFT 如此重要？

**业务痛点**：
- 全参数微调 LLaMA-7B 需要 28GB 显存、几天训练时间、几千元成本
- 大部分公司没有 A100/H100，只有消费级显卡（RTX 4090 24GB）
- 每个任务都要保存完整模型（7B × 多个任务 = 几十 GB 存储）

**PEFT 的突破**：
- 只训练 0.1%-1% 的参数
- 显存减半（28GB → 12GB）
- 训练时间缩短 10x（几天 → 几小时）
- 存储节省 1000x（14GB → 10MB）
- 效果接近全参的 90%-95%

## 📖 知识点 1：为什么需要参数高效微调 PEFT

### 核心思想

**冻结原模型，只训练少量新参数**

```
原始模型（冻结，不训练）
    ↓
  [LoRA 适配器]  ← 只训练这部分（0.1% 参数）
    ↓
微调后的输出
```

### 全参数微调 vs PEFT 对比

| 维度 | 全参数微调 | PEFT (LoRA) |
|------|-----------|-------------|
| **训练参数量** | 100%（7B 全部） | 0.1%-1%（7M-70M） |
| **显存需求** | 28GB+ | 12GB |
| **训练时间** | 几天 | 几小时 |
| **存储成本** | 每个任务 14GB | 每个任务 10MB |
| **效果** | 最好（100%） | 接近全参（90%-95%） |

### 为什么有效？

**理论依据**：
1. **内在维度假说**：模型微调时的权重变化实际上在一个低维子空间中
2. **预训练知识复用**：大模型已经学到了通用知识，微调只需要"适配"到特定任务
3. **实验验证**：研究发现权重变化矩阵 ΔW 的有效秩（effective rank）通常很小（< 100）

## 📖 知识点 2：LoRA 原理（低秩分解）

### 核心数学原理

**原始全参数微调**：
```
输入 x → [权重矩阵 W] → 输出 y

微调时：W 变成 W + ΔW
ΔW 是一个巨大的矩阵（和 W 一样大）
```

**LoRA 的改进**：
```
输入 x → [W (冻结)] + [LoRA 适配器] → 输出 y
              ↓
         ΔW = B × A
         
其中：
- W: 原始权重（冻结，不训练）
- A: 小矩阵 (d × r)
- B: 小矩阵 (r × k)
- r: 秩（rank），通常 r = 8, 16, 32
```

### 低秩分解的参数量计算

**原始矩阵**：
```
ΔW: (d × k) 矩阵
参数量 = d × k

例如：d=4096, k=4096
参数量 = 16,777,216 (1600 万)
```

**低秩分解**：
```
ΔW = B × A

A: (d × r) 矩阵 → d × r 个参数
B: (r × k) 矩阵 → r × k 个参数

总参数量 = d × r + r × k = r × (d + k)

例如：d=4096, k=4096, r=8
参数量 = 8 × (4096 + 4096) = 65,536 (6.5 万)

压缩比 = 16,777,216 / 65,536 = 256x
```

**⚠️ 关键公式（面试必考）**：
```
LoRA 参数量 = r × (d + k)
           ≠ r × d × k  ← 这是错的！

记忆口诀："A 和 B 是两个矩阵，参数量是加法不是乘法"
```

### LoRA 的完整流程

```python
class LoRALayer:
    def __init__(self, d, k, r, alpha):
        self.W = load_pretrained_weight()  # 冻结
        self.A = random_init(d, r)         # 可训练
        self.B = zero_init(r, k)           # 可训练
        self.scaling = alpha / r           # 缩放因子
    
    def forward(self, x):
        # 原始输出
        h = x @ self.W
        
        # LoRA 输出
        lora_h = (x @ self.A) @ self.B
        lora_h = lora_h * self.scaling
        
        # 合并
        return h + lora_h
```

**关键点**：
1. **W 冻结**：不计算梯度，不更新
2. **A 随机初始化**：高斯分布
3. **B 零初始化**：训练开始时 LoRA 输出为 0，不影响原模型
4. **缩放因子**：`alpha / r`，平衡 LoRA 和原模型的贡献

### 秩 r 的影响

| 秩 r | LoRA 参数量 | 压缩比 | 显存 | 适用场景 |
|------|------------|--------|------|---------|
| r=4  | 4.2M | 512x | 8 MB | 简单任务（分类） |
| r=8  | 8.4M | 256x | 16 MB | 中等任务（问答）✅ 推荐 |
| r=16 | 16.8M | 128x | 32 MB | 复杂任务（代码生成） |
| r=32 | 33.6M | 64x | 64 MB | 极复杂任务 |
| r=64 | 67.1M | 32x | 128 MB | 接近全参（少用） |

**规律**：
- r 翻倍 → 参数量翻倍 → 压缩比减半
- r 越大，效果越好，但失去 PEFT 优势
- **最佳实践**：r=8-16 是性价比最高的选择

## 📖 知识点 3：QLoRA 优化 + 超参数选择

### QLoRA = LoRA + 量化

**核心思想**：
```
原始模型（FP16/BF16，每参数 2 字节）
    ↓
量化为 4-bit（每参数 0.5 字节）← 显存减少 4x
    ↓
加载到 GPU
    ↓
加 LoRA 适配器（FP16）
    ↓
微调
```

### QLoRA 的三大核心技术

#### 1. 4-bit NormalFloat (NF4) 量化

**传统量化问题**：
- INT4：均匀分布，不适合神经网络权重（权重通常是正态分布）
- 量化误差大，效果损失明显

**NF4 的改进**：
- 针对正态分布优化的 4-bit 格式
- 量化误差更小，效果接近 FP16

**数据对比**：

| 格式 | 每参数字节 | 7B 模型显存 | 效果损失 |
|------|-----------|------------|---------|
| FP16 | 2 字节 | 14 GB | 0% (基准) |
| INT8 | 1 字节 | 7 GB | ~1% |
| INT4 | 0.5 字节 | 3.5 GB | ~3-5% |
| NF4 | 0.5 字节 | 3.5 GB | ~0.5% ✅ |

#### 2. Double Quantization（双重量化）

**问题**：量化需要存储量化常数（scaling factors），这些常数也占显存

**解决**：对量化常数再次量化

```
原始权重 (FP16)
    ↓
第一次量化 → NF4 权重 + FP16 量化常数
    ↓
第二次量化 → NF4 权重 + INT8 量化常数
```

**显存节省**：额外节省 0.4 GB（对 7B 模型）

#### 3. Paged Optimizers（分页优化器）

**问题**：优化器状态（Adam 的动量）占用大量显存

**解决**：用 CPU 内存作为"交换区"，类似操作系统的虚拟内存

```
GPU 显存不够时
    ↓
将部分优化器状态移到 CPU
    ↓
需要时再换回 GPU
```

**效果**：避免 OOM（Out of Memory），但训练速度略慢

### QLoRA vs LoRA 对比

| 维度 | LoRA | QLoRA |
|------|------|-------|
| **模型存储** | FP16 (2 字节) | NF4 (0.5 字节) |
| **显存需求** | 模型 × 2 | 模型 × 0.5 + LoRA |
| **7B 模型显存** | 12 GB | 6 GB |
| **13B 模型显存** | 18 GB | 9 GB |
| **33B 模型显存** | 45 GB | 24 GB ✅ 单卡可跑 |
| **效果损失** | 0% | ~0.5% |
| **训练速度** | 快 | 略慢（~10%） |

### 超参数选择指南

#### 1. 秩 r（Rank）

| 任务复杂度 | 推荐 r | 参数量（7B 模型） | 适用场景 |
|-----------|--------|-----------------|---------|
| 简单 | r=4 | 4.2M | 分类、情感分析 |
| 中等 | r=8 | 8.4M | 问答、摘要 ✅ 推荐 |
| 复杂 | r=16 | 16.8M | 代码生成、长文本 |
| 极复杂 | r=32 | 33.6M | 多模态、推理 |

**调参策略**：
1. 先用 r=8 做 baseline
2. 效果不够 → r=16
3. 效果还不够 → r=32（但要警惕过拟合）
4. r > 64 通常没必要

#### 2. alpha（缩放因子）

**推荐值**：alpha = 2 × r

| r | 推荐 alpha | 说明 |
|---|-----------|------|
| 4 | 8 | |
| 8 | 16 | 最常用 ✅ |
| 16 | 32 | |
| 32 | 64 | |

**作用**：
- alpha 越大 → LoRA 贡献越大
- alpha 越小 → 原模型贡献越大
- alpha/r 是实际的缩放系数

#### 3. dropout

**推荐值**：0.05 - 0.1

**调参策略**：
- 数据量大（> 10 万）→ dropout=0.05
- 数据量小（< 1 万）→ dropout=0.1
- 训练集效果好但验证集差 → 增大 dropout

#### 4. 学习率（Learning Rate）

**推荐值**：1e-4 到 5e-4

**LoRA vs 全参数微调**：
- 全参数微调：1e-5 到 5e-5（小学习率）
- LoRA：1e-4 到 5e-4（可以用更大的学习率）

**原因**：LoRA 只训练少量参数，不容易破坏原模型

#### 5. 应用 LoRA 的层

| 策略 | 应用层 | 参数量 | 效果 | 适用场景 |
|------|--------|--------|------|---------|
| 最小 | 只 Q、V | 50% | 中等 | 显存极限 |
| 标准 | Q、K、V、O | 100% | 好 | 推荐 ✅ |
| 最大 | 所有线性层 | 200%+ | 最好 | 追求极致 |

## 🔥 显存计算（面试必考）

### 全参数微调显存需求

```
总显存 = 模型 + 梯度 + 优化器 + 激活值
       ≈ 模型 × 4（粗略估算）

例如 LLaMA-13B (FP16):
13GB × 4 = 52GB
```

### LoRA 显存需求（不量化）

```
总显存 = 模型 + LoRA梯度 + LoRA优化器 + 激活值
       ≈ 模型 × 2 + LoRA × 2

例如 LLaMA-13B + LoRA (r=8):
13GB × 2 + 8.4MB × 2 ≈ 26GB
```

### QLoRA 显存需求（4-bit 量化）

```
总显存 = 模型/4 + LoRA梯度 + LoRA优化器 + 激活值
       ≈ 模型 / 4 + LoRA × 2 + 2-4GB

例如 LLaMA-13B + QLoRA (r=16):
13GB / 4 + 16.8MB × 2 + 3GB ≈ 7GB
```

### 快速判断表

| 模型大小 | 全参微调 | LoRA | QLoRA |
|---------|---------|------|-------|
| 7B | 28GB | 12GB | 6GB |
| 13B | 52GB | 26GB | 9GB |
| 33B | 132GB | 66GB | 24GB ✅ |
| 65B | 260GB | 130GB | 48GB |

**结论**：
- RTX 4090 (24GB)：最大可微调 33B 模型（QLoRA）
- A100 (80GB)：最大可微调 65B 模型（QLoRA）

## 💼 面试真题演练

### 题目 1：LoRA 原理

**面试官**："LoRA 的原理是什么？为什么低秩分解有效？"

**标准回答**：
> "LoRA 的核心思想是用两个小矩阵 A (d×r) 和 B (r×k) 的乘积来表示权重变化 ΔW，而不是直接训练一个大矩阵 (d×k)。
> 
> **参数量计算**：
> - 原始：d × k = 4096 × 4096 = 1600 万参数
> - LoRA：r × (d + k) = 8 × 8192 = 6.5 万参数
> - 压缩比：256x
> 
> **为什么有效**：
> 1. **内在维度假说**：研究发现模型微调时的权重变化实际上在一个低维子空间中，有效秩通常 < 100
> 2. **预训练知识复用**：大模型已经学到了通用知识，微调只需要调整方向，不需要改变所有参数
> 3. **实验验证**：用 0.1% 的参数就能达到全参数微调 90%-95% 的效果
> 
> **初始化策略**：
> - A 随机初始化（高斯分布）
> - B 零初始化（训练开始时 LoRA 输出为 0，不影响原模型）
> - 缩放因子 alpha/r 平衡 LoRA 和原模型的贡献"

### 题目 2：QLoRA 优化

**面试官**："QLoRA 和 LoRA 的区别是什么？为什么能用更少的显存？"

**标准回答**：
> "QLoRA = LoRA + 量化，通过三大技术节省显存：
> 
> **1. NF4 量化**：
> - 针对正态分布优化的 4-bit 格式
> - 每参数从 2 字节降到 0.5 字节（4x 压缩）
> - 效果损失只有 ~0.5%（比 INT4 的 3-5% 好得多）
> 
> **2. 双重量化**：
> - 对量化常数再次量化（FP16 → INT8）
> - 额外节省 0.4GB（对 7B 模型）
> 
> **3. 分页优化器**：
> - 用 CPU 内存作为交换区
> - 避免 OOM，但训练速度略慢 ~10%
> 
> **显存对比**（以 LLaMA-13B 为例）：
> - 全参数微调：52GB
> - LoRA：26GB
> - QLoRA：9GB（节省 5.8x）
> 
> **关键突破**：QLoRA 让 33B 模型可以在单张 RTX 4090 (24GB) 上微调，这在之前是不可能的。"

### 题目 3：超参数选择

**面试官**："LoRA 的超参数怎么选？r 和 alpha 分别是什么？"

**标准回答**：
> "**秩 r（Rank）**：
> - 控制 LoRA 的参数量和表达能力
> - 推荐值：简单任务 r=4-8，中等任务 r=8-16，复杂任务 r=16-32
> - r 越大效果越好，但失去 PEFT 优势
> - 我通常先用 r=8 做 baseline，效果不够再调到 r=16
> 
> **alpha（缩放因子）**：
> - 控制 LoRA 和原模型的贡献比例
> - 推荐值：alpha = 2 × r（如 r=8, alpha=16）
> - alpha 越大，LoRA 贡献越大
> - 实际缩放系数是 alpha/r
> 
> **其他超参数**：
> - dropout：0.05-0.1（数据少或过拟合时用 0.1）
> - learning_rate：1e-4 到 5e-4（比全参数微调大 10x）
> - target_modules：标准配置是 ['q_proj', 'k_proj', 'v_proj', 'o_proj']
> 
> **调参策略**：
> 1. 先用标准配置（r=8, alpha=16, dropout=0.05）
> 2. 效果不够 → 增大 r 到 16
> 3. 过拟合 → 增大 dropout 到 0.1
> 4. 追求极致 → 应用到所有线性层"

### 题目 4：实战场景

**面试官**："你有一张 RTX 4090 (24GB)，想微调 LLaMA-13B 做代码生成，怎么配置？"

**标准回答**：
> "首先计算显存需求：
> - LLaMA-13B 不量化：13GB × 2 ≈ 26GB → **超过 24GB，必须量化**
> - 用 QLoRA 4-bit 量化：13GB / 4 + LoRA ≈ 7GB → 可以跑
> 
> **配置选择**：
> ```python
> # 量化配置
> bnb_config = BitsAndBytesConfig(
>     load_in_4bit=True,
>     bnb_4bit_quant_type='nf4',
>     bnb_4bit_compute_dtype=torch.bfloat16,
>     bnb_4bit_use_double_quant=True,
> )
> 
> # LoRA 配置
> lora_config = LoraConfig(
>     r=16,  # 代码生成是复杂任务
>     lora_alpha=32,  # 2 × r
>     target_modules=['q_proj', 'k_proj', 'v_proj', 'o_proj'],
>     lora_dropout=0.05,
>     bias='none',
>     task_type='CAUSAL_LM'
> )
> 
> # 训练参数
> training_args = TrainingArguments(
>     per_device_train_batch_size=4,
>     gradient_accumulation_steps=4,
>     learning_rate=2e-4,  # LoRA 可以用更大的学习率
>     num_train_epochs=3,
>     fp16=True,
> )
> ```
> 
> **预期效果**：
> - 显存占用：7-9GB
> - 训练速度：batch_size=4 × gradient_accumulation=4 = 有效 batch_size 16
> - 效果：接近全参数微调的 95%
> - 训练时间：50000 条数据约 6-8 小时"

## 🔥 工程经验总结

### 真实踩坑经历

1. **坑 1：13B 模型不量化，显存爆了**
   - 问题：以为 LoRA 显存小，结果 13B 不量化需要 26GB
   - 解决：必须用 QLoRA 量化，显存降到 9GB

2. **坑 2：只对 Q、V 加 LoRA，效果很差**
   - 问题：为了省显存只对 Q、V 加 LoRA，效果比 baseline 还差
   - 解决：至少要对 Q、K、V、O 四个矩阵加 LoRA

3. **坑 3：r 设置太大，过拟合了**
   - 问题：数据只有 5000 条，用了 r=32，训练集 100% 但验证集很差
   - 解决：小数据集用 r=8，增大 dropout 到 0.1

4. **坑 4：学习率用全参数微调的值，收敛太慢**
   - 问题：用了 1e-5 的学习率，训练 10 个 epoch 还没收敛
   - 解决：LoRA 可以用更大的学习率（1e-4 到 5e-4）

### 最佳实践

1. **显存计算公式**：
   - 全参数微调：模型 × 4
   - LoRA：模型 × 2
   - QLoRA：模型 / 4 + 2-4GB

2. **标准配置**（适用 90% 场景）：
   ```python
   r=8, alpha=16, dropout=0.05, lr=2e-4
   target_modules=['q_proj', 'k_proj', 'v_proj', 'o_proj']
   ```

3. **调参顺序**：
   - 先用标准配置跑 baseline
   - 效果不够 → 增大 r（8 → 16 → 32）
   - 过拟合 → 增大 dropout（0.05 → 0.1）
   - 收敛慢 → 调整学习率（1e-4 → 5e-4）

4. **量化决策**：
   - 模型 × 2 > 显存 → 必须量化
   - 模型 × 2 < 显存 → 不量化（速度更快）
