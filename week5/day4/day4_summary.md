# 第 5 周 Day 4 深度重修：工业级复合记忆与高阶检索

> **核心目标：掌握真正的工业级 Agent 记忆架构，以及解决 RAG 召回率瓶颈的杀手锏（Multi-Query & CRAG）。**

---

## 1. 为什么不能只用 Buffer 存记忆？

**ConversationBufferMemory 的致命缺陷**：
- **Token 爆炸**：随着聊天变长，每次发送的体积越来越大，不仅极度烧钱，而且早晚一定会触发 LLM 的长文本上限截断报错。
- **注意力稀释**：文本太长会导致 LLM 失焦，出现“Lost in the Middle（中间迷失）”现象，忘记核心指令或重要的历史细节。

---

## 2. 现代 Agent 的“三层复合记忆”架构体系

真实生产环境中的 AI 助手，都会将记忆拆分为多层协同工作：

1. **短期记忆 (Working Memory)**
   - **方案**：`Sliding Window`（滑动窗口）。
   - **用法**：始终只保留最近的 N 轮对话（例如最近 3 把）。
   - **目的**：极低成本地维持当前的聊天连贯性，解决**代词指代**（例如：“那它到底该怎么弄？”）。

2. **长期记忆 (Semantic Memory)**
   - **方案**：`Vector Database`（向量数据库）。
   - **用法**：把所有历史对话全部存为向量卡片，每次用户说话时，在库里通过近似度（Embedding）检索出曾经聊过的相关内容。
   - **目的**：跨天、跨月的**长程回忆**（例如：“你还记得我上个月说我最喜欢的菜是什么吗？”）。

3. **实体档案记忆 (Entity/Summary Memory)**
   - **方案**：知识图谱或 JSON 用户画像。
   - **用法**：后台有个专门监听的小模型，每当对话中出现“我是程序员/我讨厌香菜”等永久信息时，将键值对提取并更新到画像中。

## 3. 面试大杀器：如何解决 RAG 搜不到的问题？

我们在长记忆和知识库查询时，常常遇到一个死穴：**用户提问的词汇和文档里的词汇完全对不上。**

**比如**：用户搜 `忘签到了怎么搞`，而公司的手册里写的全是 `考勤打卡异常及规范`。
传统向量检索因为词义跨度大，经常搜不出来！

### 杀手锏 1：Multi-Query（同一查询裂变展开）
在拿着用户的问题去搜数据库前，先让 LLM 把这句话“翻译、扩写”成 3 句不同的同义词搜索句。
```json
[
  "昨天忘记签到了怎么办",
  "上下班漏打卡如何处理",
  "考勤打卡异常补签流程"
]
```
拿着这 3 句话去数据库里**并行搜索**，总有一句能命中包含“打卡、考勤”的官方文档。

### 杀手锏 2：Corrective RAG (CRAG 自校正检索)
- 如果检索出来的文档根本不对路（比如查代码却查出了报销文档）。
- 系统里配置的打分裁判（Evaluator LLM）就会判定：“该文档与问题无关，触发【网络搜索工具/报错机制】”。
- 这个思想极大地**降低了 RAG 乱编瞎话的幻觉率**。
