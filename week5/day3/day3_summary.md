# 第 5 周 Day 3 深入底层重学：Function Calling 原理与兜底机制

> **核心目标：扒开 LangChain 的外衣，直面大模型真正的底层工具调用（Function Calling）生命周期。**

---

## 1. 为什么不用纯 Prompt 要求返回 JSON？

纯 Prompt 的黑客式指令（例如：“你必须只返回合法 JSON，不能包含其他话”）存在致命缺点：
1. **格式不稳定**：容易出现 `好的，这是你要的内容：{...}` 导致业务代码 `JSON.parse` 崩溃。
2. **参数瞎编（幻觉）**：私自捏造不存在的参数，导致接口调用报错。
3. **浪费资源**：需要消耗大量 Token 描述格式，且难以应对复杂参数结构。

Function Calling 是大模型底层的**硬约束提取协议**，能精准且稳定地把自然语言抽取为函数入参。

---

## 2. 完整的 3 步走工作流（面试必背！）

这是一个来回两趟（Two-Turn）的经典流程：

| 步骤 | 角色 | 动作 |
|-----|------|------|
| **第 1 步** | **用户 → LLM** | 带着问题 + `tools` (JSON Schema 函数说明书) 发起请求。 |
|      | **LLM**       | 不直接回答！只返回一个特殊的结构 `tool_calls`，里面包好了要调的函数名和装满参数的 JSON 串。 |
| **第 2 步** | **本地业务服务器**| 解析出参数，**在本地实际执行函数代码**（查 MySQL/请求第三方 API 等）。 |
| **第 3 步** | **本地 → LLM** | 使用特殊身份 `role="tool"` 将上述本地执行结果塞入对话历史，再度发给模型。 |
|      | **LLM**       | 看到数据后，最后把冷冰冰的数据“翻译总结”成有人情味的流利口语。 |

> ⚠️ **新手误区**：千万不要说“大模型去执行了代码”。大模型只负责“填参数单”，**代码永远是在你的本地业务服务器执行的！**

---

## 3. 面试高频拔高题：错误参数兜底机制

如果在第 2 步的时候，本地调用因为参数错误（比如 LLM 传了“火星”）直接抛出 `e` 异常挂掉怎么办？

**高级 Agent 的防御性编程**：
- 不要把 500 服务器错误或报错栈直接丢给用户。
- 应当 `try-catch` 包裹调用过程。
- 一旦出错，把报错信息（如：`"暂时不支持外星API"`）包装成 JSON（`{"error": str(e)}`），作为**第三步**扔回给 LLM。
- LLM 非常聪明，它看到自己的输入导致工具报错后，会自动给出圆场回答（例如：“抱歉，目前我只能查地球城市天气...”）或尝试呼叫别的工具补救！

这就是构建**长线稳定 Agent 的基石保障**！
